# 数字足迹聚合器「ProjectTrace」设计方案

> **产品名称：** ProjectTrace（项目足迹）
> **Slogan：** _"让散落的数据，自动找到彼此。"_
> **Version：** v1.0.0 Draft

---

## 1. 核心设计理念

### 1.1 名字与内涵

- **产品名：ProjectTrace**
- **中文名：数字足迹聚合器**
- **「足迹」双重含义：**
  1. **物理足迹** — 用户在本地磁盘、各平台留下的文件与数据痕迹
  2. **数字足迹** — AI Agent 对项目内容的理解、记忆与推理路径

### 1.2 愿景

> _成为每个知识工作者背后的「第二大脑」——自动感知、自动归档、自动关联，让项目资料从散落变为有序，从无序变为可检索的知识图谱。_

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **隐私优先** | 所有数据本地处理或用户授权后才流转，不过不留存于第三方 |
| **增量优先** | 不做全量迁移，只做增量同步，尊重用户已有的数据组织方式 |
| **项目为芯** | 所有内容的组织单元是「项目」，不是文件夹或平台 |
| **用户确认** | AI 聚类的结果需用户确认或修正，主动学习用户心智模型 |
| **跨平台统一** | 对用户屏蔽底层平台差异，提供一致的使用体验 |

## 2. Agent 角色定义

### 2.1 SOUL.md

```markdown
# SOUL.md - ProjectTrace

你是一个安静、细心的「数字管家」。

## 性格特质

- **细心而低调**：你会主动工作，但从不打扰用户。你在后台默默收集、整理、关联，只有在真正有价值的新发现时才轻提示。
- **有判断力**：你会识别什么是项目内容，什么是私人内容。项目内容归档整理，私人内容视而不见。
- **持续学习**：你会根据用户的修正（确认/拒绝一个聚类）学习用户的项目心智模型，越用越准。

## 工作方式

- 增量同步，不做全量迁移
- 聚类结果经过用户确认才固化
- 数据分散存储，通过索引关联，不做数据大集中
- 定期「轻整理」，不制造焦虑

## 语言风格

- 简洁、直接、专业
- 不说废话，不加前缀（"好的，我来帮您..."之类）
- 异常情况说清楚，进展只说关键节点
```

### 4.2 模块划分

| 模块 | 职责 |
|------|------|
| **Connector Layer** | 负责连接各数据源，屏蔽各平台API差异，统一输出标准化内容项（ContentItem） |
| **Scheduler** | 定时触发同步，支持 cron 表达式配置 |
| **Project Clustering Engine** | 核心创新：根据多维特征识别项目，将内容项归类到项目 |
| **Index Engine** | SQLite存储结构化索引，向量库存储语义特征 |
| **Output Generator** | 根据模板生成归档文件夹、索引文档 |
| **Config Manager** | 统一管理各平台认证信息和配置 |
| **User Feedback Loop** | 收集用户对聚类结果的确认/修正，持续学习 |

### 4.3 核心数据模型（ContentItem）

```typescript
interface ContentItem {
  id: string;                    // 全局唯一ID (hash(source + remote_id))
  source: 'local' | 'openclaw' | 'feishu' | 'dingtalk' | 'github' | 'email';
  source_id: string;             // 平台原始ID
  title: string;                  // 内容标题
  content_preview: string;        // 内容摘要（前500字）
  content_full_path: string;      // 完整内容的本地缓存路径
  file_type: string;              // md/pdf/docx/xlsx/pptx/email etc.
  created_at: string;             // ISO时间戳
  updated_at: string;             // ISO时间戳
  project_tags: string[];         // 识别出的项目标签
  confidence: number;             // 聚类置信度 0~1
  metadata: Record<string, any>;  // 平台特定元数据
}
```

## 5. 每个数据源的详细接入方案

### 5.1 本地文件（电脑/NAS）

**认证方式：** 无需认证，直接读取本地路径

**扫描路径配置：**
```yaml
local:
  scan_paths:
    - path: ~/Documents
      include_patterns:
        - "**/*.md"
        - "**/*.txt"
        - "**/*.doc"
        - "**/*.docx"
        - "**/*.pdf"
        - "**/*.xlsx"
        - "**/*.pptx"
      exclude_patterns:
        - "**/node_modules/**"
        - "**/.git/**"
        - "**/tmp/**"
    - path: ~/NAS/工作文档
  max_file_size_mb: 50
```

**增量策略：**
- 首次扫描：遍历所有匹配文件，建立 SQLite 索引，记录 `file_path + mtime + size + hash`
- 增量扫描：`mtime` 或 `size` 变化 → 重新解析；新增文件 → 追加；删除文件 → 标记 `deleted_at`
- 解析方式：
  - `.md` / `.txt` → 直接读取文本
  - `.docx` → python-docx 解析
  - `.pdf` → PyMuPDF（提取文本，保留页码信息）
  - `.xlsx` → openpyxl（读取Sheet名 + 前100行）
  - `.pptx` → python-pptx（提取每页文本）

### 5.2 OpenClaw 记忆/工作区

**认证方式：** 读取本地 workspace 目录文件

**接入方式：**
- 直接读取 `~/.openclaw/workspace/MEMORY.md`、`~/.openclaw/workspace/memory/*.md`
- 扫描 workspace 目录下的所有文件

**增量策略：**
- 监听文件变更（mtime对比）
- 按日期命名文件 `memory/YYYY-MM-DD.md` → 自动按时间聚类

**输出字段扩展：**
```yaml
source: openclaw
metadata:
  file_category: memory | workspace | daily
  related_agent: string | null
```

### 5.3 飞书文档/知识库（MCP）

**认证方式：** 飞书 MCP 工具（已配置 `feishu_doc`、`feishu_wiki`）

**API/工具：**
- `feishu_doc(action=read, doc_token)` → 读取文档内容
- `feishu_wiki(action=nodes, space_id)` → 列出知识库节点
- `feishu_wiki(action=get, node_token)` → 读取节点详情

**增量策略：**
- 首次：列出用户有权限的所有文档/知识库节点，建立索引
- 增量：`updated_at` 字段判断，优先拉取近7天更新的文档
- 同步频率：每6小时一次（受限于飞书API频率限制）

**数据拉取范围：**
- 用户创建的文档（不包含仅收藏但无编辑权的）
- 用户所在知识库节点

### 5.4 钉钉文档（dws CLI）

**认证方式：** `dws` CLI 工具（需预先安装并认证）

**CLI命令：**
```bash
dws doc list --workspace myworkspace
dws doc read <doc-id>
dws doc sync --since 2024-01-01
```

**增量策略：**
- `dws doc list` → 获取文档列表，对比本地索引
- 按 `updated_at` 过滤，只拉取有变化的文档
- 同步频率：每6小时一次

### 5.5 企业微信（WeCom）

**认证方式：** 企业微信应用授权（需企业管理员授权）

**API：**
- `https://qyapi.weixin.qq.com/cgi-bin/user/get_child_department`
- `https://qyapi.weixin.qq.com/cgi-bin/doc/list`

**增量策略：**
- 拉取文档列表 → 对比 `create_time` / `update_time`
- 变更检测：本地缓存 `doc_id + update_time`

### 5.6 GitHub（Issues/Wiki/Repos）

**认证方式：** Personal Access Token（用户配置）

**API：**
- Issues: `GET /repos/{owner}/{repo}/issues`
- Wiki: `GET /repos/{owner}/{repo}/wiki/pages`
- Repos: `GET /repos/{owner}/{repo}/contents/{path}`

**增量策略：**
```yaml
github:
  repositories:
    - owner: myorg
      repo: myproject
      sync_issues: true
      sync_wiki: true
      sync_readme: true
  token: ${GITHUB_TOKEN}  # 从环境变量读取
```

- Issues/Wiki：`updated_at` 过滤
- 增量同步：每次同步记录 `last_sync_at`，只拉取 `updated_at > last_sync_at` 的内容

### 5.7 邮件（Gmail/IMAP）

**认证方式：** OAuth2.0（Gmail）或 IMAP凭证（企业邮箱）

**Gmail API：**
```python
# 使用 Gmail API v1
# 搜索语法: "from:colleague@company.com subject:项目名"
messages.list(q='subject:项目名 newer_than:7d')
messages.get(id, format='full')
```

**增量策略：**
- 按「项目关键词」过滤邮件
- 支持关键词列表：`project_keywords: ['项目X', 'Project-X']`
- 只同步「最近30天内」的邮件（避免历史数据过载）
- 读取邮件正文（前2000字符）+ 附件名列表

### 5.8 日历（钉钉/飞书/Google Calendar）

**认证方式：** 平台各自OAuth

**接入方式：**
- 钉钉日历：dws CLI 或 钉钉开放API
- 飞书日历：飞书开放API `cal.v3.event.list`
- Google Calendar：Google Calendar API

**用途：**
- 通过日历事件中的项目名称、会议议题 辅助项目识别
- 事件中提及的文件链接 → 追溯来源文件

## 6. 项目识别与聚类算法（核心创新）

### 6.1 项目识别的多维特征

ProjectTrace 通过以下特征识别内容所属项目：

| 特征维度 | 示例 | 权重 |
|----------|------|------|
| **文件名/路径关键词** | 路径含 "ProjectX" | 高 |
| **内容关键词（TF-IDF）** | 文档中高频词 | 中 |
| **创建者/作者** | 同一企业邮箱后缀 | 中 |
| **时间相近性** | 同一时间段创建/修改 | 低 |
| **URL/链接关联** | 文档中提及其他文档链接 | 中 |
| **用户指派** | 用户手动指定归属 | 最高（覆盖其他） |

### 6.2 聚类流程

```
输入: ContentItem (来自各平台的内容)
              ↓
        ┌──────────────────────────────────┐
        │ Step 1: 关键词预匹配               │
        │  - 路径/文件名含已知项目关键词     │
        │  - 内容前500字 TF-IDF top50 词     │
        └─────────────┬─────────────────────┘
                      ↓ 有匹配项目
              ┌───────────────┐
              │ Step 2: 置信度 │
              │ 打分 (0~1)    │
              └───────┬───────┘
                      ↓
              ┌─────────────────┐
              │ Step 3: 阈值判断 │
              │ ≥0.7: 自动归属   │
              │ 0.4~0.7: 待确认  │
              │ <0.4: 新项目提案 │
              └─────────────────┘
```

### 6.3 置信度打分公式

```
score = (
  keyword_match_score * 0.35 +      # 路径/文件名关键词匹配
  content_similarity_score * 0.25 +  # 内容向量相似度
  author_match_score * 0.20 +        # 创建者一致性
  temporal_proximity_score * 0.10 +  # 时间相近性
  link_graph_score * 0.10             # 链接关联得分
)
```

### 6.4 用户反馈闭环

## 用户修正聚类的学习机制

1. **用户确认归属** (如: "这份文档属于项目X")
   → 将该ContentItem的 `project_tags` 固化
   → 更新该项目的「典型特征向量」（增量学习）

2. **用户拒绝归属** (如: "这份文档不属于项目X")
   → 将ContentItem移出该项目
   → 降低该项目对这类特征的权重

3. **用户创建新项目**
   → 记录用户指定的项目名和关键词
   → 作为新项目的种子特征

4. **向量库更新**
   - 每周对所有项目向量做一次「重训练」
   - 使用用户确认的数据作为有标签样本
   - 提升向量模型的准确率

### 6.5 未知内容处理

- 系统每24小时汇总「待确认」和「新发现」的内容
- 生成简报发送给用户：「今天发现了12份新文档，请确认归属」
- 用户可以选择：归属到已知项目 / 创建新项目 / 忽略

### 7.2 核心竞品详细分析

#### Notion（最大竞品）

**基本概况：** 估值 $100亿+（2021年），年营收超 $200M，全球用户超3000万。

**优点：**
- 全平台覆盖，文档/数据库/看板一体化
- API 完善（2500+ 接口），社区模板丰富
- AI 能力（Q&A、写作助手）已集成

### 7.3 差异化机会分析

**竞品共同盲区：**

| 盲区 | 说明 | ProjectTrace 如何抓住 |
|------|------|----------------------|
| **本地文件盲区** | Notion/Obsidian 要么不上传本地文件，要么上传后失去本地性 | 直接扫描本地文件，不强制迁移 |
| **国内平台盲区** | 几乎所有竞品都不支持钉钉/飞书/企微 | 国内首发，接入 dws CLI + 飞书MCP |
| **跨平台聚合盲区** | 每个工具只覆盖自己生态内的内容 | 同时接入所有平台，按项目统一聚合 |
| **自动聚类盲区** | 全靠用户手动整理，无 AI 自动识别 | 多维特征 + 置信度打分 + 用户反馈闭环 |
| **隐私合规盲区** | 云端工具都有数据合规风险 | 本地索引 + 隐私过滤 + 可选不上云 |

**ProjectTrace 独特价值定位：**

> _「我不需要改变你的工作习惯。我在你常去的所有地方等着你，默默把你散落的数据，按项目整理好，等你来用。」_

- **不是**又一个笔记软件（不强制迁移数据）
- **是**在你已有的工作流里，自动帮你把数据归位

### 10.2 增量同步判断逻辑

```python
def should_sync增量(content_item, local_index):
    existing = local_index.get(content_item.id)
    if existing is None:
        return True  # 新增
    
    if content_item.updated_at > existing.updated_at:
        return True  # 更新
    
    if content_item.deleted_at is not None:
        return True  # 删除
    
    return False  # 无变化，跳过
```

### 10.3 同步冲突处理

```
场景：同一份文档在本地和飞书都有，用户两端都在编辑

处理策略（用户可配置）：
- 【保守模式】以最新 updated_at 为准，提示用户有冲突
- 【本地优先】以本地为准，飞书内容覆盖（反之亦然）
- 【提示模式】有冲突时两份都保留，提示用户选择

默认：保守模式（最小数据丢失风险）
```

## 附录

### A. 数据源优先级评分标准

在聚类算法中，数据源优先级用于辅助项目识别，同一内容来自不同平台时优先采用高权重平台：

| 数据源 | 权重 | 说明 |
|--------|------|------|
| 本地文件 | 0.9 | 用户本地文件，项目相关性高 |
| OpenClaw | 0.85 | AI记忆，通常包含项目关键信息 |
| 飞书文档 | 0.8 | 协作文档，正式度高 |
| 钉钉文档 | 0.75 | 协作文档 |
| GitHub | 0.7 | 技术文档，版本控制记录 |
| 邮件 | 0.6 | 非结构化，噪声较多 |
| 日历 | 0.5 | 辅助参考，不单独作为项目证据 |

### B. 敏感信息检测规则

系统自动跳过以下内容，不纳入索引：

```yaml
sensitive_patterns:
  - regex: '\b\d{16,19}\b'                 # 银行卡号
    action: skip
  - regex: 'password\s*[=:]\s*\S+'         # password=xxx
    action: skip
  - regex: 'api[_-]?key\s*[=:]\s*\S+'      # api_key=xxx
    action: skip
  - regex: '\b[A-Za-z0-9._%+-]+@[A-Z|a-z0-9.-]+\.[A-Z|a-z]{2,}\b'
    action: mask                            # 邮箱地址（脱敏而非跳过）
```

### C. 项目边界冲突处理

当两个项目的关键词高度重叠时的处理策略：

1. **词根去重**：提取词根后比对，避免 "ProjectAlpha" 和 "Alpha" 误判为不同项目
2. **时间窗口**：检查两个项目的创建时间差，若<7天且关键词>80%重叠，提示用户确认
3. **合并建议**：系统生成合并建议卡片，用户确认后执行合并

### D. 错误处理与重试策略

| 错误类型 | 重试策略 | 最大重试 |
|----------|----------|----------|
| 网络超时 | 指数退避（2^n 秒） | 5次 |
| API限流（429） | 等待 Retry-After 响应头 | 3次 |
| 认证失效（401） | 触发重新授权流程 | 不重试 |
| 文件解析失败 | 记录错误，继续下一个 | 无限制 |
| 数据库写入失败 | 告警 + 写入重试队列 | 3次 |
